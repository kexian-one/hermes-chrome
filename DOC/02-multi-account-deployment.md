# 02 — 多机 × 多账号物理部署

**20+ 个 1688 买家账号,跑在 3-4 台 PC 上,每台 6 账号,每台有人值守。**

## 为什么"6 账号/机 × 多机",而不是"1 账号/机 × N 机"或"20 账号/机 × 1 机"

| 方案 | 优点 | 缺点 | 判定 |
|---|---|---|---|
| 1 账号/机 × 20 机 | 完全隔离 | 硬件成本高 / 人工值守不现实 | ❌ |
| 20 账号/机 × 1 机 | 节省硬件 | 单机内存/CPU 爆 / 单机一挂全停 | ❌ |
| **6 账号/机 × 3-4 机** | 内存可控 / 1 人值守可行 / 分布式容错 | 需要 N 套代理和值守 | ✅ |

**单机 6 账号的资源占用估算**:
- 内存: 6 × ~400MB Chromium = ~2.5GB(加 OS / 其他 ~5-6GB,16GB 机够用)
- CPU: 浏览器空闲时低,集中点击 / 收消息时中等(8 核以上不卡)
- 网络: 6 个独立代理出口 — 看代理质量
- 显示器: 6 窗口 1080p 摆不下,**部分要最小化或多桌面**

## 为什么用"6 个不同浏览器"而不是"同浏览器 6 个 profile"

**这是关键决策**。两个方案对比:

| 维度 | 同浏览器 6 profile | **6 个不同浏览器**(选用) |
|---|---|---|
| User-Agent | 6 个一样(都是 Chrome 同版本) | 6 个不同(Chrome / Edge / Brave 等) |
| Canvas / WebGL 指纹 | 基本一致 | 浏览器内核版本不同 → 有差异 |
| 字体表 / 浏览器特有 API | 一致 | 不同浏览器有差异 |
| 进程隔离 | 共享父进程 | 完全独立进程 |
| 任务栏视觉区分 | 全是 Chrome 图标 | 不同图标,**一眼能区分** |
| 1688 风控关联难度 | 中(主要靠登录态 / 代理) | **更难**(指纹也不一样) |
| 扩展安装 | 装 1 次,各 profile 共享 | **要在 6 个浏览器各装 1 次** |
| 启动配置 | `--user-data-dir` 一行命令 | 每浏览器命令行参数略不同 |
| 单浏览器挂 | 该 profile 挂,其他 profile 不影响(同进程下) | 该浏览器挂,完全不影响其他 |

**用户决策选择"6 个不同浏览器"**。理由(用户原话整理):
- 不同浏览器内核 → 风控指纹差异最大
- 6 个独立进程 → 真正物理隔离
- 视觉上一眼能区分(任务栏 / Alt-Tab 不会混)

## 6 个浏览器的具体选型建议

候选(都是 Chromium 内核 = 都能装 Chrome Store 扩展):

| # | 浏览器 | 特点 | 备注 |
|---|---|---|---|
| 1 | Chrome | 基准 | Chrome Store 扩展直接装 |
| 2 | Edge | 微软,Win 自带 | 先打开"允许其他商店的扩展" |
| 3 | Brave | 反指纹追踪自带,**默认更隐私** | 直接装 Chrome 扩展 |
| 4 | Vivaldi | 高度可定制 | 直接装 Chrome 扩展 |
| 5 | Opera | 自带 VPN(可选) | 直接装 Chrome 扩展 |
| 6 | 360 极速 / 搜狗 / QQ 浏览器 | 国产 Chromium | 部分支持,需测扩展是否能装 |

**也可以全选国外浏览器**(更安全的指纹组合):Chrome / Edge / Brave / Vivaldi / Opera / Arc(Mac/Win 都有版本)

**注意**:
- Arc / Opera GX 等部分浏览器**对扩展支持有限制**,接入前必须实测能否安装 `open-claude-in-chrome`
- 国产浏览器(360/搜狗/QQ)安装 Chrome Store 扩展可能要走非常规路径(如开发者模式 + 离线 crx),稳定性需测

(已包含在上节"6 个浏览器选型建议")

## 浏览器选型:全 Chromium 内核即可

**用户决策**:不限于 Chrome,只要 Chromium 内核就行:
- Chrome
- Edge(支持装 Chrome Store 扩展)
- Brave(默认支持 + 自带反指纹)
- Vivaldi / Opera 等

**好处**:不同浏览器自带不同指纹(User-Agent / Canvas hash 等),进一步降低 1688 跨账号关联风险。比"同浏览器多 profile"更隔离。

**注意点**:
- Edge / Brave 等非 Chrome 安装 `open-claude-in-chrome` 扩展可能需要打开"允许其他应用商店扩展"
- Native Messaging Host(如果扩展用到)在每个浏览器的注册路径不同:
  - Chrome: `%LOCALAPPDATA%\Google\Chrome\NativeMessagingHosts\`
  - Edge: `%LOCALAPPDATA%\Microsoft\Edge\NativeMessagingHosts\`
  - Brave: `%LOCALAPPDATA%\BraveSoftware\Brave-Browser\NativeMessagingHosts\`

## IP 代理设计

**每个账号必须独立 IP**。原因:
- 同 IP 多账号 = 1688 风控关联 ≈ 100%
- 即使浏览器指纹不同,IP 相同就关联

**代理类型**:

| 类型 | 评价 |
|---|---|
| **住宅代理**(BrightData / Smartproxy / Soax / 国内供应商) | ✅ 推荐,IP 池纯净,被识别为机房风险低 |
| 机房代理 | ❌ 1688 一眼能识别为代理(ASN),反而是黑名单 |
| 数据中心代理 | ❌ 同上 |
| 手机 4G 拨号 | △ 可行但维护麻烦,IP 池小 |

**配置方式**:
```
chrome.exe --user-data-dir="D:\Profiles\acct-A1" --proxy-server="http://代理IP:port"
```

或用扩展(SwitchyOmega 等)在 profile 内单独配代理。

**预算估算(20+ IP)**:
- 国外住宅代理:**$50-200/月**(20 个静态 IP)
- 国内住宅代理:**¥300-1500/月**
- 详见 `08-tech-stack-and-costs.md`

## 滑块人工值守

**策略**:每台机 1 个人坐前面,看到滑块就拖。

**辅助机制**:
- **微信 / 钉钉 push** 提醒(人离座时手机响)
- **本机响铃** + 浏览器窗口标题加 🔴 标记
- **不做** 集中状态 dashboard(因为人就在现场,直接看屏幕够)

**预期工作量**:
- 一次完整 batch 全网约 200-400 次滑块
- 平摊每机每 batch 50-100 次
- 单次拖 5-15 秒
- 平均频率:**每 2-5 分钟一次**(2-3 小时 batch 时段内)

**人员要求**:
- 不需要技术背景,**会用电脑 + 看屏幕能识别滑块就行**
- 一个人盯一台机即可

## 任务分配策略

20+ 账号怎么把任务分给各机?

### 简单策略(MVP):静态分账号

```
PC1: 账号 A1-A6 → 任务标签 "machine:PC1" 或不打标签默认本机
PC2: 账号 B1-B6 → 任务标签 "machine:PC2"
PC3: 账号 C1-C6 → 任务标签 "machine:PC3"
```

任务派发时,**任务里明写 account=A3,自然就跑在 PC1**(因为 A3 属于 PC1)。
账号 → 机器映射写死在配置文件里。

### 复杂策略(后期):动态负载均衡

- 全部任务进队列,不绑死机器
- 每台机有"账号池",空闲时拉任务
- 滑块卡的账号自动跳过
- 等中央调度成熟再加,**MVP 不做**

## 单点故障与恢复

**每台机自治** = 单机挂不影响其他机器。

| 故障 | 影响 | 恢复 |
|---|---|---|
| 单账号触滑块 | 该账号暂停 | 人工拖,自动恢复 |
| 单账号 cookie 失效 / 强制退登 | 该账号不可用 | 提醒人重新登录 |
| 单浏览器挂 | 该账号不可用 | 启动脚本自动重启 |
| 单代理 IP 失效 | 该账号不可用 | 换 IP / 报警 |
| 整机断电 / 网络挂 | 该机 6 账号停 | 重启机,任务自动重派(中央队列保留) |
| Claude Code / Agent runtime 崩溃 | 该机停 | 重启,浏览器仍登录 |
| 飞书 / 任务队列服务挂 | 全网新任务无法派发 | 已派发的不影响 |

## 启动脚本示例

一键启 6 个 Chromium(以 PC1 为例):

```powershell
# D:\multi-1688-chase\start-pc1.ps1

$accounts = @(
    @{name="A1"; port=17861; proxy="http://代理IP_A1:8080"},
    @{name="A2"; port=17862; proxy="http://代理IP_A2:8080"},
    @{name="A3"; port=17863; proxy="http://代理IP_A3:8080"},
    @{name="A4"; port=17864; proxy="http://代理IP_A4:8080"},
    @{name="A5"; port=17865; proxy="http://代理IP_A5:8080"},
    @{name="A6"; port=17866; proxy="http://代理IP_A6:8080"}
)

foreach ($acc in $accounts) {
    $profile = "D:\Profiles\acct-$($acc.name)"
    $exe = "C:\Program Files\Google\Chrome\Application\chrome.exe"  # 或 Edge / Brave
    # 启动时通过 open-claude-in-chrome 设置端口的方式取决于扩展实现
    Start-Process -FilePath $exe -ArgumentList @(
        "--user-data-dir=$profile",
        "--proxy-server=$($acc.proxy)",
        "--window-name=Account-$($acc.name)"
    )
    Start-Sleep -Seconds 2  # 给浏览器启动时间
}

# 启动 aggregator
python D:\multi-1688-chase\aggregator\main.py --config D:\multi-1688-chase\config\pc1.yaml
```

## 物理摆放建议

每机:
- 主显示器 1 个,1080p+ 即可
- 6 浏览器窗口 — 用虚拟桌面(Win10/11 自带,Ctrl+Win+方向)各放一个或几个
- 微信 / 钉钉在另一台手机或副屏(收 push)
- 不需要鼠标键盘自动化设备 — **真鼠标 = 软件 `mcp__open-claude-in-chrome__computer` 命令,不是物理鼠标**
