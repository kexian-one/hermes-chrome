# 00 — 业务背景、目标、设计约束

## 业务背景

### 起点:1688 发票催单

- 用户是 1688 平台的**重度买家**(720+ 待开发票订单,横跨 130+ 家供应商)
- 月底/季末需要批量催商家开票,人工逐个找订单 → 找卖家 → 旺旺发消息,**极其耗时**
- 当前已通过 Claude Code + `open-claude-in-chrome` MCP 扩展实现了 **fapiao-1688**(抓数据)和 **fapiao-1688-chase**(催单)两个 skill,可工作但**只能跑单账号**

### 规模诉求

- 实际有 **20+ 个 1688 买家账号**(可能是不同主体的分账号 / 不同业务线)
- 期望:**所有账号统一调度,集中催单 + 集中跟进**
- 分布:用户准备 **3-4 台 PC**,每台跑 **6 个账号**(原因见 `02-multi-account-deployment.md`)

### 长期诉求(不止 1688 催单)

用户预期这只是**第一个落地业务**。未来同一个 Agent 平台还要:

- 通过**飞书 / 钉钉**接其他类型的任务(LLM 可解析自然语言)
- 处理类似但**不同流程**的事(如 1688 跟订单进度 / 退款 / 客服 / 询价等)
- **共享知识库**:所有 agent 能看到同一份"行业知识"+"踩过的坑"
- **经验自动沉淀**:每天/每周自动把跑过的任务总结成可复用的知识
- **远程安装新 skill**:不重新部署就能给 agent 加能力

## 目标(分层)

### 短期目标(本月内)

- **业务效果验证**:把已发的 batch1+2 的催单跟踪到结果,算出开票率提升和工时节约
- **决策依据形成**:基于业务效果决定要不要投入下面的工程

### 中期目标(2-3 个月)

- **MVP 自建 Agent**:单机版,接管 1688 催单这一个场景,摆脱 Claude Code 依赖
- **多机多账号上线**:3-4 台机, 18-24 账号正常运营
- **任务多元化**:接入第 2-3 个任务类型(不限于催单)

### 长期目标(6-12 个月)

- **平台化**:同一套 Agent 跑多个业务领域(发票 / 订单 / 客服等)
- **知识沉淀**:平台用得越久,agent 越聪明 / 越省成本
- **可分享 skill**:把 skill 做成可分发的资产(plugin / git 仓库等)

## 设计约束(硬约束,不可妥协)

| # | 约束 | 来源 / 原因 |
|---|---|---|
| C1 | 必须用 `open-claude-in-chrome`(开源 MCP 扩展) | Playwright 已实测被 1688 风控,真 Chrome + 扩展才认 |
| C2 | LLM 必须用 OpenAI 兼容 API | 用户明确不用 Anthropic Claude,目标 provider(GPT/DeepSeek/Qwen/GLM)都兼容 |
| C3 | 所选 LLM 必须支持多模态输入 | 用户明确选型时只考虑多模态模型 |
| C4 | 滑块处理走人工 | 打码服务有合规风险 + 1688 反打码,人工更可控 |
| C5 | 不依赖 Claude Code(长期) | 用户明确要自建,不绑死 Anthropic 工具链 |
| C6 | 多账号必须 IP 隔离 | 同 IP 多账号 100% 被 1688 风控关联 |
| C7 | 单浏览器 / 扩展不能跨账号 | Chrome profile = 登录态隔离,必须每账号独立 profile |
| C8 | 单机最多 6 浏览器实例 | 用户硬件假设,内存/CPU 不爆 |

## 设计软约束(优先级降低但有偏好)

| # | 软约束 | 偏好原因 |
|---|---|---|
| S1 | 倾向国产 LLM(DeepSeek / Qwen / GLM)优先 | 成本低 + 合规风险小 |
| S2 | 倾向 Python(LLM 生态完整) | 用户/团队大概率 Python 熟 |
| S3 | 任务接入倾向飞书 / 钉钉 | 用户已使用的 IM |
| S4 | 一切 skill / knowledge 文件化(可 git) | 版本控制 + 跨机同步友好 |
| S5 | 单机自治(不强依赖中央服务) | 减少运维负担,某机挂不影响其他 |

## 风险地图

主要风险按"业务、风控、技术、运营"四个维度:

### 业务风险

- **R-B1**: 催单效果有限(商家不回 / 不开票) → 投入工程量浪费
  - 缓解:先用现有 skill 跑 batch1+2,数据说话再投入

### 风控风险

- **R-K1**: 1688 风控升级,扩展被识别封号
  - 缓解:实时监控滑块频次,异常立刻人工接管
- **R-K2**: 同主体多账号被关联识别
  - 缓解:不同账号用不同浏览器 / 代理 IP / 不同操作节奏
- **R-K3**: 打码 / 自动操作触发 1688 法务关注
  - 缓解:不用打码,全人工拖

### 技术风险

- **R-T1**: `open-claude-in-chrome` 不支持多端口 / 多实例
  - 缓解:fork 源码改,见 `03-aggregator-design.md`
- **R-T2**: LLM provider 切换时 tool calling 兼容性差异
  - 缓解:OpenAI 兼容大同小异,做兼容性测试
- **R-T3**: 多模态视觉模型对 1688 UI 识别准确率不高
  - 缓解:Vision 主要做兜底,主路径走 JS DOM 操作

### 运营风险

- **R-O1**: 滑块频次过高,值守人员撑不住
  - 缓解:节奏控制,减慢任务投递率
- **R-O2**: 代理 IP 不稳定,任务大量失败
  - 缓解:选住宅代理 + 自动健康检查
- **R-O3**: 任意机器挂掉导致整批任务卡住
  - 缓解:每机独立,挂了不影响其他

## 不在范围(本文档不讨论)

- 1688 商家端的 ERP 集成(只做买家侧)
- 法务 / 合规审查(假设用户已自行评估,自动化操作仅限自己账号)
- 票据 OCR / 入账系统对接(发票收到后的处理流程)
- 跨平台扩展(淘宝 / 京东 / 拼多多 等,虽然架构可复用)
