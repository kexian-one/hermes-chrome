# 07 — 分阶段实施路线

## 总体策略

**分 5 个 Phase 推进,每个 Phase 是可独立验证的里程碑**。不要试图一次性把所有东西建好。

每个 Phase 结束都要 demo + 评估:**值不值得继续投入下一个 Phase**。

## Phase 0: 验证业务效果(当前阶段)

**目标**:用现有 Claude Code + skill 跑完一轮催单,**用数据证明这事有 ROI**。

| 任务 | 工时 | 产出 |
|---|---|---|
| Batch1 + Batch2 已发 21 家(已完成)→ 跟踪开票情况 | 1 周观察 | 开票率 / 时效报告 |
| Batch2 剩余 50 家发完 | 1.5-2 小时(单账号) | Batch2 全部 65 家发完 |
| 一周后再抓数据对比 | 5 分钟 | 开票完成率 |
| 整理业务效益(节省工时 / 开票率提升 / 金额回收) | 1 小时 | 一份决策报告 |

**Go/No-Go 标准**:
- 开票率 > 70%(满意,继续 Phase 1)
- 开票率 50-70%(可接受,Phase 1 可以小步上)
- 开票率 < 50%(效果不佳,先调整文案,再评估)

**这是整个 roadmap 的"质保签"** — 数据不好就别投后面的工程。

## Phase 1: 单机单账号 MVP(自建 Agent 雏形)

**目标**:**摆脱 Claude Code 依赖**,用自建 Python 进程完成一家催单。

| 模块 | 工时 |
|---|---|
| 单 OpenAI compat adapter | 0.5-1 天 |
| 简易 LLM Loop(无 router,固定 1 个 provider) | 1-2 天 |
| Skill loader(读 Claude Code 格式的 SKILL.md) | 1 天 |
| Aggregator MCP server 雏形(只接 1 个 open-claude-in-chrome) | 1-2 天 |
| 命令行任务输入 + 控制台输出 | 0.5 天 |
| 联调 + 测试 | 1-2 天 |
| **小计** | **5-8 天** |

**Demo 标准**:用户在命令行说"催 X 商家的发票",自建 Agent 完成催单,跟之前用 Claude Code 跑结果一致。

**风险点**:
- LLM tool calling 稳定性可能不如 Claude → 提前测 DeepSeek / Qwen 在 chase 场景的实际表现
- open-claude-in-chrome 直接 MCP 连接的细节(端口、握手等)需要摸清楚

## Phase 2: 单机多账号 + Aggregator

**目标**:单台 PC 上跑 6 个不同浏览器 + 6 账号,Aggregator 统一管理。

| 模块 | 工时 |
|---|---|
| Aggregator 多 client 支持 + 账号路由表 | 1-2 天 |
| 多浏览器启动脚本(6 个 Chromium 浏览器,各自配置) | 1 天 |
| open-claude-in-chrome 多浏览器端口配置(需先考察,见 09) | 0.5-2 天 |
| 6 个住宅代理对接 + 健康检查 | 1 天 |
| 滑块感知 + 暂停/恢复 + IM push | 1.5-2 天 |
| 任务 routing(按 account 分配) | 0.5 天 |
| 高层工具封装(chase_send_one 等) | 1.5-2 天 |
| 测试 + 调优 | 2-3 天 |
| **小计** | **9-13 天** |

**Demo 标准**:6 个账号并行发催单,触发滑块时 IM push 提醒,人工拖完自动恢复。

**风险点**:
- open-claude-in-chrome 不支持多端口 → 需 fork 源码改(0.5-2 天工作量)
- 6 浏览器同时跑,内存 / CPU 压力(老机器可能撑不住)
- 6 账号同 PC 风控阈值是否会降(实测)

## Phase 3: 多机协调 + 共享 Git

**目标**:扩到 3-4 台 PC,共享 skill + knowledge,跨机任务派发。

| 模块 | 工时 |
|---|---|
| 第二台机器部署(复用 Phase 2 的代码) | 0.5-1 天 |
| Git 仓库布局(skills/, knowledge/, memory/) | 0.5 天 |
| SessionStart hook 自动 git pull | 0.5 天 |
| 任务队列(Redis Streams) | 1-2 天 |
| 飞书 / 钉钉机器人接入 → 任务派发 | 2-3 天 |
| 任务 routing(按 machine tag 分配) | 1 天 |
| 简易 Web Dashboard(看每台机状态) | 2-3 天 |
| 测试 + 调优 | 2-3 天 |
| **小计** | **9-14 天** |

**Demo 标准**:在飞书里说"催湖北达利的发票",PC1 自动接到任务,完成后飞书反馈。

## Phase 4: 经验沉淀 + 知识库

**目标**:每天 / 每周自动从 task logs 萃取经验,沉淀到 knowledge/,所有 agent 共享。

| 模块 | 工时 |
|---|---|
| Task log 标准化(jsonl 格式) | 1 天 |
| Experience Extractor LLM pipeline | 2-3 天 |
| Knowledge merger(合并到已有 knowledge,处理冲突) | 1-2 天 |
| Knowledge 注入到 LLM prompt(全量 / 检索) | 1 天 |
| 向量检索(可选,如果 knowledge > 100 文件再加) | 1-2 天 |
| Cron 调度 + 定时跑 | 0.5 天 |
| 测试 + 调优(确保萃取出的知识有用,不是噪音) | 2-3 天 |
| **小计** | **8-12 天** |

**Demo 标准**:跑一周后,knowledge/ 目录自动出现新的 markdown 文件(总结的规律),下次任务的 LLM prompt 会引用这些规律。

## Phase 5: 任务多元化 + 平台化

**目标**:不只催发票。开始接其他业务场景。

| 任务类型 | 工时 | 产出 |
|---|---|---|
| 新增 skill 模板(SKILL.md 标准化) | 1 天 | 团队能写新 skill 不踩坑 |
| 第二个业务场景接入(如订单跟进 / 退款催办) | 3-5 天 | 第二个 skill 上线 |
| Skill 自动安装机制(git push 即可)| 0.5 天 | 团队成员可以自助加 skill |
| Web Dashboard 完善(任务可视化 / 干预 / 历史) | 3-5 天 | 产品级管理面板 |
| 监控告警(账号挂、限流、错误率) | 2-3 天 | 运维友好 |
| 文档 + 团队培训 | 2-3 天 | 团队能上手 |
| **小计** | **11-17 天** |

**Demo 标准**:用户能在飞书发"帮我跟进 12345 订单",自建 Agent 走 order-tracking skill 完成任务并报告。

## 推荐节奏

**单人** | 每个 Phase 1-2 周 | 总耗时 3-4 个月

**2-3 人小组** | 每个 Phase 0.5-1 周 | 总耗时 6-10 周

**关键依赖**:
- Phase 0 业务效果是 go/no-go gate
- Phase 1 → 2 → 3 是工程递进,每步 demo 才进下一步
- Phase 4 + 5 可以并行(Phase 4 是后端能力,Phase 5 是业务扩展)

## 跳级风险评估

| 跳级 | 后果 |
|---|---|
| 跳 Phase 0 直接做工程 | 没业务数据,投入风险高,可能做完没人用 |
| 跳 Phase 1 直接 Phase 2 | 单账号问题没暴露,多账号会一起爆 |
| 跳 Phase 3 直接 Phase 4 | 没多机数据,经验沉淀没素材 |
| 跳 Phase 4 直接 Phase 5 | 新业务跑起来没规律可学,每次都从零 |

**建议:逐 Phase 推进,每个 Phase 完成都强制 demo 评估**。

## 各 Phase 的"放弃"信号

如果在某个 Phase 看到这些信号,**及时止损**:

| Phase | 放弃信号 |
|---|---|
| 0 | 业务效果差(< 30% 开票率提升)+ 看不到改进方法 |
| 1 | LLM 在 chase 场景表现远不如 Claude(失败率 > 30%)→ 考虑混用 |
| 2 | open-claude-in-chrome 不支持多端口且 fork 难度大 → 找替代扩展或换路 |
| 3 | 多机同步问题大、数据不一致 → 简化为单机持久部署 |
| 4 | 经验沉淀产出的"知识"实际没用 → 简化为人工写 knowledge |
| 5 | 新业务场景需要的能力扩展太大 → 评估每个新业务的 ROI |

## 团队 vs 单人推进

**单人推进**(用户自己写代码):
- 时间长(3-4 个月)
- 但每一步深度理解,没有人传递成本
- 适合:你是技术负责人,愿意亲自上手
- 适合:不急,慢工出细活

**2-3 人小团队**(找工程师协助):
- 快(6-10 周)
- 需要写清楚需求文档(本目录就是)+ 阶段性 review
- 适合:你不熟 Python / 想快速验证 / 业务紧迫
- 适合:已经看到 Phase 0 效果好,值得多投入

## 接入策略(团队工程师视角)

如果你找了工程师,推荐的接入方式:

1. **读完本目录 00-09 全部 markdown**(总共 1-2 小时)
2. **跑一遍现有 fapiao-1688 + fapiao-1688-chase skill**(感受现状,半天)
3. **从 Phase 1 开始,严格按本路线推进**
4. **每个 Phase 提交一次 demo + 简短报告**(让你 review)
5. **遇到`09-open-questions-and-todos.md` 里没答案的事**,先停下来问你
